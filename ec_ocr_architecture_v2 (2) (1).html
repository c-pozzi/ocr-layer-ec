EC OCR Pipeline v2 — Tiered Architecture

Classifier-routed processing with Qwen 7B Q4 for page classification, then tiered GPU workers for OCR.

1. Architecture Overview

The key insight: use a cheap classifier pass on each page to decide the processing tier. Most pages are straightforward (single column, typed text, clean scan) and go through the fast path. Complex pages (tables, multi-column, handwritten, blurry) route to bigger instances with specialized prompts or few-shot examples.

2. The Classifier Stage

A small quantized Qwen runs a short classification prompt on each page before any heavy OCR processing happens.

What the classifier detects

• multi_column — two or more text columns

• has_tables — structured tables with rows/cols

• handwritten — handwritten text or annotations

• poor_quality — blur, fading, stains

• has_stamps — official stamps or seals

• has_forms — form fields to fill

• language — if not in metadata already

Routing rules

Tier 1 (simple): none of the complex flags set

Tier 2 (complex): any of these are true:

• has_tables == true

• handwritten == true

• poor_quality == true

• multi_column == true AND has_tables == true

The routing logic is just a function — easy to tune thresholds and rules without changing any infrastructure.

# Classifier prompt — minimal, ~50 tokens input
CLASSIFY_PROMPT = """Output JSON only, no explanation:
{"multi_column": bool, "has_tables": bool, "handwritten": bool,
"poor_quality": bool, "has_stamps": bool, "has_forms": bool}"""

# Routing function
def route_page(classification: dict) -> str:
    if classification["has_tables"]:
        return "complex"
    if classification["handwritten"]:
        return "complex"
    if classification["poor_quality"]:
        return "complex"
    return "simple"

# Prompt selection based on classification
def select_prompt(classification: dict) -> str:
    if classification["has_tables"]:
        return TABLE_OCR_PROMPT # includes few-shot table examples
    if classification["handwritten"]:
        return HANDWRITTEN_OCR_PROMPT
    if classification["poor_quality"]:
        return ENHANCED_OCR_PROMPT # "try harder" instructions
    return STANDARD_OCR_PROMPT
Why per-page, not per-document? A single document can have page 1 as clean typed text, page 5 as a table, and page 12 as a handwritten annotation. Per-page routing ensures each page gets the right treatment.

Instance sizing for Tier 2: Qwen 72B in FP16 requires ~144GB VRAM for weights alone. With KV cache for few-shot examples (2-4 image-text pairs at 2000-4000 tokens), total VRAM can reach 200GB+. The p4d.24xlarge provides 320GB across 8× A100 40GB GPUs with NVLink, giving ample headroom for tensor parallelism and large batch sizes.

4. Detailed Processing Flow

Step 1: Orchestrator Lambda (22:00 UTC)

Scans s3://ec-ocr/incoming/ for new PDFs.

Sends one SQS message per document to the Raw Queue.

Requests classifier instances via AWS Batch.

Step 2: Classifier Worker

Pulls doc from Raw Queue → downloads PDF from S3.

Extracts all pages as images.

Runs Qwen 7B Q4 classify prompt on each page.

For each page, writes classification JSON to S3 and sends to the appropriate tier queue (simple or complex) with the page S3 path + selected prompt.

Step 3a: Tier 1 Workers

Pull from Simple Queue.

Download page image from S3.

Run Qwen 7B FP16 with standard OCR prompt.

Write page result to s3://ec-ocr/results/{doc_id}/page_{n}.json

Fast, cheap, handles the bulk of the work.

Step 3b: Tier 2 Workers

Pull from Complex Queue.

Download page image + few-shot examples from S3.

Run Qwen 72B FP16 with tensor parallelism across 8× A100 GPUs, using specialized prompts.

Write page result to same path structure.

Slower, more expensive, but better quality on hard pages.

Step 4: Assembler

When all pages for a document are done (tracked in DynamoDB), an Assembler Lambda fires.

Collects all page JSONs → combines into full document result.

Writes final output to s3://ec-ocr/results/{doc_id}/full.json

Updates DynamoDB status to completed.

Step 5: Shutdown

Tier 1 workers shut down when Simple Queue is empty for N minutes.

Tier 2 workers shut down when Complex Queue is empty.

Classifier shuts down when Raw Queue is empty.

Everything scales to zero. Cost = $0 when idle.

5. Key Design Decision: Where Does the Classifier Run?

Three options for where to run the Q4 classifier:

Option How it works Pros Cons
A. Separate classifier instance Dedicated g5.xlarge runs only Q4 classifier. Routes pages to tier queues. Clean separation. Classifier can process all docs fast before OCR starts. Extra instance cost. Extra S3 round-trips.
B. Co-located on Tier 1 workers Each Tier 1 worker loads both Q4 and FP16 models. Classifies then OCRs simple pages locally, only sends complex pages to Tier 2 queue. Fewer instances. No extra S3 round-trip for simple pages (~80%). Both models fit in 24GB (Q4 ~4GB + FP16 ~14GB = ~18GB). More complex worker code. Less VRAM headroom for batching.
C. CPU-based pre-filter + Q4 on GPU Use CV2 blur detection and layout heuristics on Lambda (CPU). Only ambiguous cases go to Q4 classifier. Cheapest. Many features detectable without VLM. Less accurate routing. Two-system maintenance.
Recommended: Option B. The classifier (Qwen 7B Q4, ~4GB) and the Tier 1 OCR model (Qwen 7B FP16, ~14GB) both fit on a single A10G (~18GB total, tested). This means 70-80% of pages are classified and processed on the same machine with no extra network hop. Only complex pages (20-30%) are routed via SQS to the Tier 2 p4d.24xlarge running 72B.

6. AWS GPU Pricing Models

Pricing Model How It Works Discount vs On-Demand Risk / Tradeoff
On-Demand Pay per hour, no commitment. Start and stop anytime. 0% (baseline price) None. Always available, but most expensive.
Spot Bid for unused AWS capacity. AWS can reclaim instances with 2-minute warning. Up to 60-90% for common instances. For high-demand GPU instances (p4d), realistic discount is 40-60%. Interruption risk. Not guaranteed to be available. Works well for batch workloads with SQS retry — if a worker is reclaimed, the message reappears in the queue.
Reserved (1yr / 3yr) Commit to a fixed instance type for 1 or 3 years. Pay upfront or monthly. 30-60% depending on term and payment option Lock-in. Must pay whether the instance is used or not. Only makes sense for always-on workloads, not batch.
Savings Plans Commit to a $/hr spend (not a specific instance type). Flexible across instance families. ~30-50% More flexible than Reserved, but still a commitment. Better if workload might change instance types.
Capacity Blocks Reserve GPU instances for a defined time window (e.g., 3 hours daily). Relatively new AWS offering for ML workloads. Varies, but guaranteed availability at a known price Must plan schedule in advance. Good fit for predictable batch jobs like this pipeline.
Recommendation for this pipeline: Tier 1 (g5.xlarge) works well on Spot — these are widely available and cheap. Tier 2 (p4d.24xlarge) is harder to get on Spot due to high demand. Options: (1) Use Spot with on-demand fallback — try Spot first, fall back to on-demand if unavailable. (2) Use Capacity Blocks if the batch schedule is predictable (e.g., every night at 22:00 for ~3 hours). (3) For sustained daily usage, a Savings Plan may be more cost-effective than pure on-demand.

The batch scale-to-zero pattern is critical here. Running p4d.24xlarge for ~3 hours/day instead of 24/7 reduces monthly cost from ~$23,900 to ~$1,450-2,950 depending on pricing model. This is why the architecture is designed around queues and auto-shutdown.

7. Cost Estimate (Tiered)

Daily Load: ~300 docs × ~30 pages = ~9,000 pages

Classification pass: ~0.3 sec/page × 9,000 = ~45 min GPU time (if separate) — included in Tier 1 if co-located

Tier 1 (simple, ~7,000 pages):

7,000 pages × 1.4 sec = ~2.7 hrs GPU time

g5.xlarge Spot: 2.7 hrs × $0.50 = $1.35/day

Tier 2 (complex, ~2,000 pages):

With 72B FP16 on p4d.24xlarge: 2,000 pages × 5 sec = ~2.8 hrs

p4d.24xlarge On-Demand: 2.8 hrs × $32.77 = ~$92/day

p4d.24xlarge Spot (~50% off): 2.8 hrs × $16 = ~$45/day

Monthly estimate: ~$1,400/month (Spot) to ~$2,800/month (On-Demand) for Tier 1 + Tier 2

Plus ~$10-20/month for S3, SQS, Lambda, DynamoDB

The tiered approach saves money: the p4d.24xlarge is only needed for 20-30% of pages and only runs for ~3 hours/day. Compared to running p4d.24xlarge 24/7 (~$23,900/month), the tiered batch approach at ~$1,400-2,800/month is a 88-94% saving. Scale-to-zero is what makes GPU-heavy pipelines financially viable.

8. Docker Image Strategy

Tier 1 Image (co-located classifier)

ec-ocr-tier1:latest

Contains: vLLM + Qwen 7B Q4 weights + Qwen 7B FP16 weights

Image size: ~25GB

Startup: loads both models into VRAM (~18GB)

Worker logic: classify → route or OCR locally

Tier 2 Image

ec-ocr-tier2:latest

Contains: vLLM + Qwen 72B FP16 weights + few-shot example bank

Image size: ~150GB (72B weights are large)

Startup: loads model sharded across 8× A100 with tensor parallelism

Worker logic: pull from Complex Queue, run specialized prompt

Few-shot example bank: Store few-shot examples (image + expected output pairs) in S3: s3://ec-ocr/few-shot/tables/, s3://ec-ocr/few-shot/handwritten/, etc. Workers download them at startup. This allows updating examples without rebuilding the Docker image.
